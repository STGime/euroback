Ten days, four user-facing changes — including the one a lot of you have been waiting for: versioned schema migrations for your own project. Everything below is already live; no action needed on your side.

🧱 Versioned schema migrations for your project

Until now, table changes through the console were one-offs — fine for a sketch, painful for a team. The CLI now ships a real migration channel for your tenant schema:

eurobase migrations new add_orders_table eurobase migrations up eurobase migrations status

Numbered SQL files under ./migrations, applied transactionally, recorded per-project. Each migration runs under a short-lived per-tenant role that can only see your own schema — a malicious RESET ROLE in a migration body can't pivot into another project.

Honest footnote: shipping this channel cost us three deploy retries against managed Postgres (default INHERIT / SET options differ from local PG). It's solid now, but if you tried migrations up on June 12 or 13 and saw a 503 — it works today, please retry.

⚡ Edge functions can finally be real webhook receivers

Custom headers are forwarded

Functions now receive your caller's request headers — X-Signature, X-Api-Key, X-Webhook-*, the works. Stripe, GitHub, Twilio and the rest now have everything they need to verify a request. The gateway still withholds two families on purpose: your Eurobase auth credentials (so a function can't impersonate the caller upstream) and the X-Eurobase-* control namespace (so a caller can't spoof a signed identity header).

Query string is preserved

new URL(req.url).searchParams now actually contains what the caller sent. The one exception: a caller-supplied ?apikey= is stripped before reaching your function, mirroring how we already stripped Authorization — a caller's secret should never end up in your function's logs.

🚨 Breach register & 24h/72h SLA workflow (plumbing first)

For the GDPR-conscious among you: the platform now has a real incident channel. Open an incident, identify affected subjects from your own data, fire the 24h customer notification (recipients resolved server-side from your project members — no caller-supplied list, since "official-looking breach notice to arbitrary inbox" is a footgun), then the 72h authority notification. All timed against the SLA, all written to an append-only register and the audit log. MTTD and MTTR are exported as metrics.

Honest scope: the API and runbook ship today (/compliance/breaches/* + docs/runbooks/breach-notification.md). The console UI for it is next — we wanted the audited, append-only register and the SLA timers to be in place first, so by the time you click "Open incident" the clock and the paper trail are already correct. If you want to wire it up to your own on-call now, the endpoints are documented; if not, the console workflow lands in the next update.

🛠 CLI docs caught up with the CLI

The command reference had drifted. It's now accurate:

  • The full edge-functions group (list, deploy, get, invoke, logs, delete) is documented — it was missing entirely.
  • version, projects delete, db create-table / add-column / drop-column / drop-table, and compliance report / sub-processors are now listed.
  • The platform-only admin migrations commands are out of the user docs — they ran against a local dev DB, not your project, and listing them just invited confusion.

Update with brew upgrade eurobase, then eurobase --help to see the real list.
