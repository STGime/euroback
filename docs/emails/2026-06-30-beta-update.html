📤 Bring your own SMTP provider

Until now, every auth email (verification, password reset, magic link) went through the shared Eurobase sender. Fine for a sketch, painful past a few signups a day. You can now plug in your own provider and Eurobase routes your project's auth mail through it — your sender domain, your deliverability, your provider's quota.

Where: Project → Auth → SMTP tab. Fill in host / port / username / password / from-address / encryption (STARTTLS, TLS or none). Hit Save, then Send test to verify before the project starts using it — the upstream's exact error is shown if it fails, so a misconfig dies at setup, not at first signup.

Your SMTP password is sealed at rest with your project's per-tenant key (same AES-256-GCM pipeline as your vault secrets and edge-function env vars). The API never returns it, and an "edit other fields without retyping" affordance keeps the saved one when you leave the password blank on an edit.

If you point at a US provider (SendGrid, Mailgun, Postmark, SES, Mandrill, SparkPost, SMTP.com), the console shows an amber advisory: your email content leaves the EU jurisdiction even though the rest of your project doesn't. It's advisory, not a block — your data, your call. EU alternatives worth knowing if you want sovereignty end-to-end: Scaleway TEM, Brevo (FR), Mailjet (EU), Mailtrap EU.

⚡ Per-project rate limits

Five auth knobs + an "IP Address Forwarding" toggle, all on one page. Cap signup/sign-in rate per IP, token-refresh rate, token-verify rate, emails/hour, SMS/hour. Defaults are platform-wide and sane for a project up to a few thousand signups/day; you only touch the page if you want to tighten or loosen a specific knob.

Where: Project → Auth → Rate Limits tab. Empty input = platform default (the placeholder shows the current default value). Over-quota responses arrive as HTTP 429 with a Retry-After header — your SDK can surface a friendly message and retry cleanly.

If your app sits behind a CDN or reverse proxy you control, flip the IP Address Forwarding toggle so the limiter keys off the leftmost X-Forwarded-For entry instead of the TCP peer. Leave it off otherwise — a forged XFF header would dodge the limit.

📦 One-click DSAR exports (GDPR Article 15 + 20)

When a user emails "what do you have on me?" or a customer asks for their data so they can leave, the Compliance → Data Export tab now handles both in one click. Per-user export grabs every row in your tenant tables that references their user_id, plus the auth record. Full project export bundles every table + the auth manifest + the storage manifest + the audit log as a single zip. Pick JSON for round-trippable structure or CSV for spreadsheets.

The 30-day GDPR Article 12 deadline stays statutory; the tooling stops being the bottleneck. All bytes stay on Scaleway fr-par, download links expire after 7 days, and every request / completion / failure is recorded in your Audit Log with the actor's email + IP — your evidence trail is built in, not bolted on.

Pricing note: the one-click console flow is part of the Pro tier from now on. The underlying export API endpoints stay open on Free — DSAR is a legal obligation and we won't paywall a path to GDPR compliance. If you're on Free and an authority knocks, you can still meet the deadline by scripting the export against the API. Pro saves you from writing the script. Either way: the Audit Log, rate limiting, and EU residency are identical.

🐛 Edge functions: "Function not found" bug fixed

If you tried to open an edge function in the console over the past week or two and saw a red banner saying "Function not found" — that was a backend bug masking the real reason. Two issues actually: (a) the gateway was collapsing every error from the function read into a 404, hiding the actual cause, and (b) the underlying cause was a SQL scan that didn't handle NULL on the new sealed-env-vars column for functions created before the encryption migration shipped. Both are fixed. Open any function in the console and you'll land in the editor.
