✉️ Email signup, verification, reset, magic-link — all fixed

What was broken: if you turned on require_email_confirmation, or triggered a password-reset or magic-link email, the link the user received pointed at a Eurobase console URL that didn't exist. Result: user gets the email, clicks the link, hits a 404, and can never confirm. Every downstream email flow was silently dead. If you'd tried this and given up — sorry, it's on us.

What's live now: you configure your own tenant URLs and the emails link to those. Three new fields on the Auth Settings tab, one per flow:

  • Email verification URL — where signup emails point; your page reads ?token=… and calls verifyEmail(token).
  • Password reset URL — your page reads the token and calls resetPassword(token, newPassword).
  • Magic-link URL — your page reads the token and calls verifyEmail(token).

Each URL must appear in your Redirect URLs allowlist — same open-redirect defence Supabase runs. The console checks this inline; if you toggle require_email_confirmation on without a verification URL, Save is blocked with a helpful message rather than silently breaking signup.

Per-call override lands too: pass emailRedirectTo on signUp / requestMagicLink / forgotPassword / resendVerification to point one specific email somewhere else (e.g. a staging domain during QA). Same allowlist check applies.

📦 SDK 0.5.0 — four new methods for the email flow

Signup used to be a one-way door: your user got the email, and you had no way to finish the flow client-side. The SDK now ships every completion method the tenant guide walks through:

  • verifyEmail(token) — completes signup or a magic-link sign-in.
  • forgotPassword(email, {emailRedirectTo?}) — kicks off the reset flow. Always resolves to {error: null}, no email enumeration.
  • resetPassword(token, newPassword) — finishes the reset from your tenant page.
  • resendVerification(email, {emailRedirectTo?}) — sends a fresh confirmation. Same anti-enumeration shape as forgot-password.

npm i @eurobase/sdk@0.5.0. Changelog + worked ?token=… examples in the README.

🚧 Supabase → Eurobase migration — on the way

A migration path from Supabase is in the works. It'll cover the four surfaces that matter — database, auth users, storage, edge functions — as a CLI that reads your Supabase project, translates each piece to Eurobase's conventions, and hands you a plan you review before applying. Nothing runs against your target project without your say-so.

Not ready to use yet. We're still validating it end-to-end against real projects. When it's genuinely ready, we'll say so in a beta update — and if you're interested in helping test once we're at that stage, reply and I'll flag you when the invitation goes out.
